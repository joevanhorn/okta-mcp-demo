Securing AI Agent Access with Okta

Identity-Governed Entitlements for Enterprise AI Agents via the Okta MCP Adapter
Platform: Okta Workforce Identity + OIG Protocol: Model Context Protocol (MCP)

The Challenge

Enterprise teams are adopting AI agents — Claude Cowork, Claude Code, GitHub Copilot, Glean — to automate workflows across internal systems. These agents connect to backend resources (Salesforce, ServiceNow, Jira, internal APIs) through MCP, an open protocol for agent-to-tool communication.

MCP standardizes how agents discover and invoke tools, but it has no native identity or authorization layer. Without an identity-aware control point, enterprises face three gaps:

🔓 No Per-User Authorization

Agents connect with static API keys or service accounts. Every user gets the same access regardless of role, department, or need. There is no least-privilege enforcement.

👻 No Visibility

IT and Security have no audit trail of which user invoked which tool through which agent. Agent activity is invisible to existing SIEM and compliance infrastructure.

🔑 Credential Sprawl

API keys and personal access tokens proliferate across agent configurations. Offboarded employees' credentials may persist in agent configs with no lifecycle management.

⚖️ No Governance

Agent entitlements exist outside IAM governance. No access request workflows, no periodic reviews, no time-bound access, no separation of duties.

Architecture

The Okta MCP Adapter acts as a secure intermediary between AI agents and backend systems. It authenticates users through Okta, scopes their access based on identity policy, and proxies tool calls to backend MCP servers with a per-user token. The backend server reads the token scopes and only exposes tools the user is entitled to.

Entitlement Model

Agent tool access is governed through Okta group membership. Each group maps to a set of OAuth scopes, and each scope maps to a tier of tools on the backend MCP server. Users only see tools matching their assigned scopes.

Okta Group Token Scopes Tools Available Example Actions
Cowork-CRM-Read sfdc:read Search, view opportunities and accounts "Show me open opportunities for Acme Corp"
Cowork-CRM-Write sfdc:read sfdc:write All read tools + create and update opportunities "Create a new opportunity for the Q3 renewal"
Cowork-ITSM-Read snow:read Search, view incidents "Show me all P1 incidents from this week"
Cowork-ITSM-Write snow:read snow:write All read tools + create and update incidents "Create an incident for the VPN outage"
(no Cowork groups) (no resource scopes) No tools visible to the agent Agent authenticates but cannot take any action
ℹ️
Zero Standing Privilege Users authenticate successfully through Okta but receive zero tool access until explicitly granted an entitlement. The agent cannot discover, invoke, or even attempt to call tools outside the user's scoped entitlements.

Governance with Okta Identity Governance

Because agent entitlements are modeled as Okta group memberships, they are immediately governable through OIG's existing workflows — no custom connector or resource type development required.

Access Requests

Users request agent entitlements through the OIG self-service portal. Each entitlement tier is a catalog item with a configured approval chain. Time-bound access is supported — elevated entitlements (e.g., write access) can auto-revoke after a defined period.

Request Type Grants Approval Duration
Cowork: CRM Read Access Cowork-CRM-Read Manager Persistent
Cowork: CRM Write Access Cowork-CRM-Write Manager + App Owner 8 hours (default)
Cowork: ITSM Read Access Cowork-ITSM-Read Manager Persistent
Cowork: ITSM Write Access Cowork-ITSM-Write Manager + App Owner 8 hours (default)

Access Certifications

Periodic certification campaigns review agent entitlements alongside traditional application access. Reviewers see each user's Cowork group memberships and can confirm or revoke. Revocation removes the group membership, and the user's agent tools disappear on next connection.

Considerations

Enterprise deployment of AI agents with governed access requires awareness of the current state of the ecosystem. The following considerations reflect the maturity of both the Okta MCP Adapter and the broader agent platform landscape as of April 2026.

🔄
Session Lifecycle and Token Refresh Users must re-authenticate when their token expires or when their entitlements change. Entitlement changes (group adds/removes via OIG) take effect on next authentication, not mid-session. The adapter's short-lived tokens (configurable TTL) limit the window of stale entitlements.
🌐
Remote Agent Connectivity Cloud-based agents (e.g., Claude Cowork) connect to MCP servers through their platform's cloud infrastructure, not from the user's machine. The adapter must be internet-reachable and must correctly handle the PKCE redirect flow through the agent platform's proxy. Validate this flow during deployment for each agent platform.
📊
Observability Integration Agent activity through the adapter is logged in the adapter's own audit log. Today, this is separate from both Okta System Log and the agent platform's native telemetry (e.g., Anthropic's OpenTelemetry). SIEM streaming is on the adapter roadmap to unify these into a single observability pipeline. Until then, correlating agent activity across layers requires manual log aggregation.
🏗️
Backend MCP Server Responsibility The adapter scopes the token; the backend MCP server must read the scopes and filter tools accordingly. In this demo, the MCP server also integrates Okta FGA checks — every tool invocation is evaluated against the FGA relationship store before execution, enforcing per-resource authorization. Vendor-provided MCP servers (e.g., from Salesforce or ServiceNow) may not support scope-based tool filtering or FGA integration today. Custom backend MCP servers provide full control over both behaviors.
📋
Agent Platform Maturity Anthropic's Claude Enterprise provides SCIM-based user provisioning, role-based access controls, per-tool connector restrictions (org-wide), and OpenTelemetry event streaming. However, Cowork activity is currently excluded from Anthropic's Audit Logs, Compliance API, and Data Exports. The adapter's audit trail provides a compensating control for this gap. Evaluate the agent platform's governance maturity alongside the Okta integration when planning deployment scope.
🔒
Tenant Restrictions Claude Enterprise supports tenant restrictions via proxy header injection (anthropic-allowed-org-ids) to prevent users from bypassing organizational controls with personal accounts. This is an Enterprise-only feature and requires network proxy configuration (Zscaler, Prisma, Netskope, etc.). Without tenant restrictions, users can switch to personal accounts on the same machine and bypass org-level guardrails.
🎯
Granularity Trade-offs The current model governs entitlements at the tier level (read, write, admin) using Okta groups. Per-tool granularity is architecturally possible with OIG custom entitlement definitions but adds governance overhead. For most enterprises, tier-based entitlements provide the right balance of security and operational simplicity. Okta FGA is now live in the adapter, providing fine-grained, per-resource authorization at invocation time — checking whether a user can act on a specific account or record — without requiring per-tool OIG entitlements. The adapter's OPA policy engine (on roadmap) will add context-aware per-tool authorization within the adapter layer itself.

Per-Tool Entitlement Governance — Future Architecture

The tier-based model (read / write / admin) covers most enterprise use cases. For organizations requiring per-tool governance through OIG, the following architecture describes what individual tool entitlement management would require.

What It Would Look Like

Each MCP tool becomes a discrete OIG entitlement on an application resource. Instead of a user holding "CRM Write Access" as a group membership, they would hold individual entitlements like sfdc:create_opportunity, sfdc:update_opportunity, and sfdc:log_activity. These would appear as certifiable entitlements in OIG campaigns and requestable items in the access catalog.

What It Would Require

Component Requirement
OIG Resource Model Register the MCP Adapter (or each backend) as an OIG-managed application with fine-grained entitlements defined per tool
Entitlement Sync A SCIM connector or governance connector to push tool-level entitlement definitions into OIG and sync assignments back to the authorization layer
Token Claims Auth server would need to issue tokens with per-tool claims rather than tier-level scopes, or the adapter would need to resolve entitlements from a policy store at invocation time
Adapter Policy Engine OPA or FGA integration in the adapter to evaluate per-tool authorization decisions at runtime (on the adapter roadmap)
Certification UX Reviewers would need to understand individual tool semantics during certification. For a user with 15 tools across 3 backends, this may create reviewer fatigue. Bundle-based certification (certify the tier, not each tool) may be more practical.
⚠️
Recommendation Start with tier-based entitlements. Monitor actual usage patterns through the adapter's audit log. If specific tools require tighter governance (e.g., bulk_export or delete operations), selectively add fine-grained controls for those tools rather than governing every tool individually.

Resource-Level Authorization with Okta FGA

The previous sections address two authorization questions: "which tools can this user see?" (scope-based filtering) and "which individual tools should be governed?" (per-tool OIG entitlements). Okta Fine-Grained Authorization (FGA) addresses a third, fundamentally different question: "can this user perform this action on this specific resource?"

✅
FGA Is Live In this demo, the MCP server checks Okta FGA on every tool invocation. Product team members can read all accounts but cannot invoke write tools. Sales reps can only modify accounts in their territory. The FGA store contains 67 relationship tuples covering users, accounts, territories, and tool permissions.

Three Layers of Authorization

Layer Question Answered Mechanism Evaluation Time
1. Authentication + Scoping
(Adapter)		 Who is this user, and which backends can they reach? PKCE + MFA → scoped OAuth token based on Okta group membership At session start
2. Tool Filtering
(Backend MCP Server)		 Which tools can this user see and invoke? Backend reads token scopes, filters tools/list response At tool discovery
3. Resource Authorization
(Okta FGA)		 Can this user perform this action on this specific resource? Relationship-based check against Okta FGA policy store (OpenFGA / Zanzibar model) At every tool invocation

When FGA Matters

Consider a sales representative with Cowork-CRM-Write access. The tier-based model grants them the create_opportunity tool. But should they be able to create opportunities for any account? In most Salesforce orgs, reps own specific accounts or territories.

Okta FGA models these relationships: user:jsmith has relationship owner on account:acme-corp, which grants permission can_create_opportunity. When the agent invokes create_opportunity for Acme Corp, the backend MCP server checks FGA before executing the API call.

FGA Advantages Over Scope-Only Authorization

⚡ Real-Time Revocation

Scope-based authorization is checked at token issuance. If entitlements change mid-session, the old scopes persist until re-authentication. FGA is checked on every invocation — revocation is effectively instant.

🎯 Resource Specificity

OAuth scopes cannot express "write access to Acme Corp's opportunities but not Global Industries'." FGA's relationship model can. The authorization decision accounts for the specific resource being acted upon, not just the action category.

🔗 Relationship Inheritance

FGA supports relationship hierarchies. A user who owns an account can automatically create_opportunity for that account. If the account belongs to a territory, the territory manager inherits access. These relationships compose naturally without requiring explicit per-resource entitlements.

📐 Open Standard

Okta FGA is built on OpenFGA, an open-source implementation of the Google Zanzibar authorization model. The same relationship model can be used across agent authorization, application authorization, and data access control — one policy framework for the enterprise.

Prerequisites for FGA Integration

Prerequisite Details
Authorization Model Define the FGA type definitions: which resource types exist (accounts, opportunities, configuration items, etc.), which relationships are valid (owner, member, viewer), and which permissions derive from those relationships
Relationship Data Populate the FGA store with relationship tuples synced from source systems. For Salesforce: account ownership and territory assignments. For ServiceNow: business unit membership and CI ownership. This sync must be ongoing, not one-time.
Backend Integration Backend MCP server must call the FGA Check API before executing write operations, passing the user identity, the action, and the target resource. Read operations may or may not require FGA checks depending on data sensitivity.
Adapter Policy Engine The adapter's OPA/FGA policy engine (on roadmap) would centralize these checks in the adapter layer rather than requiring each backend to integrate independently
ℹ️
Phased Adoption FGA adds the most value for write operations on sensitive resources — creating, modifying, or deleting records where the user's relationship to the specific resource matters. Read operations and low-risk writes can remain scope-governed. In this demo, FGA is live with 67 tuples covering territory-based account ownership and role-based tool permissions. Start with the tier-based model, layer in FGA for high-impact operations as the relationship data becomes available.
